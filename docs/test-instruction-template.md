# AIエージェント向け テストコード作成指示テンプレート v1

---

## 0. 【タスク固有設定】← タスク生成時に必ず埋めること

> ここが未記入の場合、このテンプレートは使用しないこと。

```yaml
# --- 対象スコープ ---
target_files:
  - "{対象ファイルパス1}"
  - "{対象ファイルパス2}"

target_functions:
  - "{対象クラス/関数/APIエンドポイント1}"
  - "{対象クラス/関数/APIエンドポイント2}"  # 最大2つまで

test_scope:
  include: "{今回追加すべきテストの範囲を具体的に記述}"
  exclude: "{今回は対象外にする観点（例: パフォーマンス、並行性）}"

# --- 仕様との紐付け ---
source_spec: "{仕様書ファイルパスまたはURL}"  # 例: docs/spec.md#section-3
ac_ids:
  - "{AC-01: 受け入れ条件の内容を一行で記述}"
  - "{AC-02: 受け入れ条件の内容を一行で記述}"

# --- プロジェクト設定 ---
language: "{使用言語}"
test_framework: "{使用フレームワーク}"
mock_library: "{使用モックライブラリ}"
coverage_tool: "{使用カバレッジツール}"
coverage_threshold: {目標カバレッジ %}

# --- タスク粒度制約 ---
task_constraints:
  max_test_cases: 12        # 1タスクで追加するテストの上限
  min_test_cases: 5         # 1タスクで追加するテストの下限
  refactor_policy: "テスト成立に必要な最小限の修正のみ許可"
```

---

## 1. 基本原則

- **テストファースト**：実装の前にテストを書き、テストが失敗（RED）することを確認してから実装に入ること
- **仕様の証明**：テストは「コードが動く」ことではなく「仕様を満たしている」ことを証明するものとして書くこと
- **独立性の確保**：各テストは他のテストに依存せず、単独で実行可能であること
- **スコープの厳守**：セクション0の `target_files` / `target_functions` 以外のコードには原則触れないこと。対象外機能のリファクタリングは禁止

---

## 2. テスト観点チェックリスト（フェーズ別優先度）

セクション0の `test_scope.exclude` に記載された観点は対象外とする。

### Phase 0（必須）：すべてのタスクで対応
- [ ] 正常系：代表的な入力値での期待動作
- [ ] 主要な異常系：不正入力・必須欠落・例外スロー
- [ ] 副作用：DB・外部ストアの状態が正しく変化すること

### Phase 1（原則対応）：特別な理由がない限り対応
- [ ] 境界値：最大・最小・空・null・0・false
- [ ] エラーメッセージの内容が仕様通りであること
- [ ] 外部API呼び出しの引数・回数が正しいこと（モック検証）

### Phase 2（必要に応じて対応）：セクション0の `test_scope.include` に明示された場合のみ
- [ ] 冪等性：同一入力で複数回実行しても結果が変わらないこと
- [ ] 並行実行・競合状態への耐性
- [ ] パフォーマンス上限
- [ ] 異常系の連鎖（複数エラーが同時に起きた場合）

## 3. テストコードの記述ルール

### 3-1. 仕様との紐付け（必須）
各テストケースには、セクション0の `ac_ids` から対応するIDを必ずコメントで明記すること。
**ACIDが存在しない観点のテストは追加しないこと（コメント捏造禁止）。**

```javascript
// AC-01: 在庫が0の場合、注文は受け付けられない
test('在庫0のとき注文するとエラーになる', () => { ... })
```

### 3-2. アサーションの品質
- 1テストに対してアサーションは原則1〜3個に絞り、何を検証しているかを明確にすること
- `toBeTruthy()` のような曖昧なアサーションは避け、具体的な値を検証すること
- 異常系ではエラーの型・メッセージ・コードも検証対象に含めること

### 3-3. テストの命名規則
以下のいずれかの形式で統一すること。

```
// 形式A: 状況_操作_期待結果
'在庫が0のとき_注文を実行すると_エラーが返る'

// 形式B: 自然言語で仕様を表現
'在庫が0の場合は注文できない'
```

### 3-4. テストの独立性
- `beforeEach` / `afterEach` で状態を初期化し、テスト間の依存を排除すること
- グローバル変数・シングルトンの状態はテストごとにリセットすること
- テストの実行順序を変えても結果が変わらないこと

### 3-5. モック・スタブの使用方針

以下の方針で層ごとに使い分けること。全部モックにしないこと。

| 依存の種類 | 方針 | 理由 |
|---|---|---|
| 外部APIクライアント | モック化（呼び出し引数・回数を検証） | ネットワーク依存を排除 |
| リポジトリ層（DB） | インメモリ偽実装を使用 | 統合上の壊れ方を拾うため |
| 時刻・乱数 | 固定値に差し替え | 再現性確保のため |
| 同一サービス内の別クラス | 原則モックしない | 過剰なモックは統合バグを見逃す |

モックを使用した箇所はコメントで理由を明記すること。

## 4. 自己検証ステップ（実装後に必ず実行）

```
Step 1. テストがグリーン（PASS）であることを確認する
Step 2. 実装の核となるロジックを意図的に壊し、テストがレッド（FAIL）になることを確認する
Step 3. 壊した実装を元に戻し、再度グリーンになることを確認する
Step 4. Step 2でレッドにならなかったテストは、検証内容を見直して修正する
```

> ⚠️ Step 2でテストがレッドにならない場合、そのテストは仕様を証明していないと見なす

---

## 5. 終了条件

以下をすべて満たした場合にタスク完了とする。

- [ ] すべてのテストがグリーンであること
- [ ] テスト観点（セクション2 Phase 0・1）の全項目がカバーされていること
- [ ] 追加テストケース数が `min_test_cases` 以上 `max_test_cases` 以下であること
  - ⚠️ 件数達成のために薄いテストを追加することは禁止。件数より内容を優先すること
- [ ] コードカバレッジが `coverage_threshold` % 以上であること
- [ ] セクション0の `ac_ids` に列挙されたすべての受け入れ条件に対応するテストが存在すること
- [ ] 自己検証ステップ（セクション4）をすべて完了していること
- [ ] 各テストにACIDコメントが記載されていること
- [ ] `target_files` / `target_functions` 以外のコードを変更していないこと

---

## 6. 禁止事項

| 禁止パターン | 理由 | 代替手段 |
|---|---|---|
| テストのためだけに実装にフラグを追加する | テスト都合で仕様が歪む | 依存注入・モックで対応 |
| `expect(true).toBe(true)` 等の常に通るアサーション | 仕様を何も検証しない | 具体的な値を検証する |
| テスト間で状態を共有する | 実行順依存のフレイキーなテストになる | beforeEach で初期化 |
| 実装コードとテストコードを同時に書いて一発でグリーンにする | REDを確認しないと検証力が保証できない | TDDの手順に従う |
| 全ケースを1つのテストに詰め込む | 失敗時の原因特定が困難になる | 1テスト1観点に分割 |
| 外部依存をすべてモックにする | 統合上の壊れ方を拾えなくなる | セクション3-5の方針に従う |
| ACIDのないテストにそれらしいコメントを捏造する | 仕様との紐付けが虚偽になる | 仕様外の観点はスコープ外とする |
| スコープ外ファイルのリファクタリング | タスク範囲が際限なく広がる | 必要なら別タスクとして起票する |
